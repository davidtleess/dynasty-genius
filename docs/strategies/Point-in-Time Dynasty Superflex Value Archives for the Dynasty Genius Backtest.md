# Point-in-Time Dynasty Superflex Value Archives for the Dynasty Genius Backtest

## TL;DR
- **The single best integrity-passing source is the DynastyProcess open-data GitHub repo (`github.com/dynastyprocess/data`, file `files/values.csv`).** Its weekly GitHub Actions commit history gives genuine, immutable, git-dated point-in-time snapshots that include a Superflex value column (`value_2qb`), and all four target dates (Sept 2021–2024) are almost certainly recoverable by checking out the commit nearest each date. **Critical caveat: these are FantasyPros-ECR-derived values, NOT the trade-market values your forward FantasyCalc collection uses** — so they test "does the model beat expert consensus," not "beat the FantasyCalc market."
- **FantasyCalc — your preferred source — has no usable public history.** Its API (`api.fantasycalc.com/values/current`) is current-only with no documented date parameter; the only path to 2021–2024 FantasyCalc Superflex values is fragile Wayback Machine captures of the JSON endpoint, which must be checked capture-by-capture and may not exist for all four dates.
- **Recommended plan:** Use DynastyProcess git history as the spine of the backtest (it passes all four integrity tests), clearly label it as the source, and treat its ECR-derived nature as a known methodology caveat; in parallel, probe Wayback for FantasyCalc JSON captures near each date and the Dynasty Daddy KTC database for a market-based cross-check. Skip any date you cannot recover cleanly rather than substituting a weak source.

## Key Findings

**1. DynastyProcess `data` repo passes all four point-in-time integrity tests — with one methodology caveat.**
- **Superflex present:** The `values.csv` schema includes `value_2qb` (the Superflex/2-QB value) alongside `value_1qb`, `ecr_2qb`, `ecr_1qb`, `ecr_pos`, `age`, `draft_year`, `player`, `pos`, `team`, `fp_id`, and `scrape_date`. This is confirmed verbatim from the official `ffscrapr::dp_values()` reference (ffscrapr.ffverse.com), which shows a "tibble: 747 × 12" with columns `player pos team age draft_year ecr_1qb ecr_2qb ecr_pos value_1qb … value_2qb <int>, scrape_date <chr>, fp_id <chr>`.
- **Dating is intrinsic/verifiable (two independent stamps):** The file is regenerated weekly by a GitHub Action and committed, so every snapshot carries (a) an immutable git commit date and (b) an in-file `scrape_date` column. This is exactly the "git commit date" provenance you wanted, and the two stamps cross-check each other.
- **Survivorship passes by construction:** Each weekly commit overwrites `values.csv` with that week's then-current FantasyPros ECR. Checking out an old commit returns the file exactly as it stood that week — including veterans valued then who have since retired. Old commits are immutable, so there is no back-revision or survivorship-scrubbing risk.
- **History reaches 2021:** The repo has 2,326 commits; secondary documentation snapshots show DynastyProcess values content "last updated on 2021/09/09," i.e., live and being committed right at the first target date.
- **The caveat:** DynastyProcess values are *not* a trade market. They are generated by scraping FantasyPros Dynasty Expert Consensus Ranks (ECR) and converting along an exponential-decay curve — DynastyProcess states the formula verbatim as "**Value = 10500 * e^(FP ECR*-0.0235)**" — with the 2QB values produced by a LOESS conversion of 1QB→2QB ECR. The league assumption is also FantasyPros': per DynastyProcess, "This chart printout assumes what FantasyPros assumes: a typical 12 team PPR league starting QB/RB/RB/WR/WR/TE/FLEX/FLEX rostering about 300 players." Your forward pipeline collects FantasyCalc (a real-trade market). Using DynastyProcess as the historical "market" therefore tests the model against expert consensus, not against the same market signal you collect going forward.

**2. Key/ID format requires a crosswalk.** `values.csv` is keyed by `fp_id` (FantasyPros ID), not `sleeper_id`. The repo's own `db_playerids.csv` provides the crosswalk (`fantasypros_id` → `sleeper_id`, `mfl_id`, `gsis_id`, `espn_id`, etc.). Because you build on Sleeper IDs, you'll join `values.csv.fp_id` → `db_playerids.csv.fantasypros_id` → `sleeper_id` locally.

**3. FantasyCalc has no historical endpoint.** The documented API is `https://api.fantasycalc.com/values/current?isDynasty=true&numQbs=2&numTeams=12&ppr=1`. The `/values/current` path returns only the present snapshot; no date/asOf parameter is documented, and no paid historical tier surfaced in research. Encouragingly, the *current* player object does include both `sleeperId` and `mflId` (the FantasyCalc guest post by Josh Cordell shows e.g. `'name': 'Justin Jefferson', 'mflId': '14836', 'sleeperId': '6794', … 'value': 10726`), so current FantasyCalc data crosswalks cleanly — but that doesn't help for 2021–2024. The only FantasyCalc history path is Wayback Machine captures of the API JSON, which are unverified and likely sparse.

**4. KeepTradeCut has no public API or official history; Dynasty Daddy is a possible KTC-history proxy.** KTC publishes current Superflex dynasty values only. Dynasty Daddy (`github.com/G-Sher/dynasty-daddy`) scrapes KTC into a Postgres database and ties players to Sleeper IDs — its README states: "This database is populated using a python cron job that scrapes KeepTradeCut's player rankings once a day (since KeepTradeCut currently doesn't have a public api). For Sleeper's data, we use sleeper's public apis." It has operated since roughly 2021 and exposes "value over time" features. If its database retains historical daily rows back to 2021, it would be a genuine, Sleeper-keyed, market-based (crowdsourced KTC) point-in-time SF source — but historical-row retention back to the target dates was not confirmed in research and must be verified.

**5. License:** DynastyProcess `data` is GPL-3.0 (a project by Tan Ho and Joe Sydlowski). Local, non-redistributed analytical use for a personal backtest is unproblematic; flag that GPL terms attach if you redistribute the data or derivative datasets. FantasyCalc/KTC data are proprietary/crowdsourced; local personal use is generally tolerated but redistribution is discouraged.

### Exact retrieval steps for the top source (DynastyProcess)

1. `git clone https://github.com/dynastyprocess/data.git && cd data`
2. For each target date, find the nearest commit that touched the values file:
   `git log --until="2021-09-15" -1 --format="%H %cI" -- files/values.csv` (repeat with 2022/2023/2024). Confirm the returned commit date is within ±7 days of the Sept 8 target.
3. Extract that snapshot without disturbing your working tree:
   `git show <COMMIT_SHA>:files/values.csv > values_2021-09.csv` (repeat per date).
4. Extract the matching crosswalk at the same commit (so IDs line up to that era):
   `git show <COMMIT_SHA>:files/db_playerids.csv > ids_2021-09.csv`.
5. Locally join `values.*.csv` (`fp_id`) → `db_playerids.*.csv` (`fantasypros_id`) to attach `sleeper_id`; keep `value_2qb`, `pos`, `player`, and `scrape_date`.
6. **Validate point-in-time integrity:** confirm the in-file `scrape_date` matches the commit date (a later `scrape_date` would indicate a revised row — reject it), and confirm since-retired veterans appear at their then-current values.
7. **Fallback for very old commits:** if `files/values.csv` 404s at a 2021 SHA, list the tree at that commit (`git ls-tree <SHA> files/`) and try `files/values-players.csv`, `files/db_values.csv`, or an `archives/` path. (Per the repo README, "A number of older files were moved into the archives/ folder," so an older file name is plausible at 2021 commits.)

## Recommendations
1. **Build the backtest spine on DynastyProcess git history now.** Run retrieval steps 1–6 for all four dates. It is the only source that simultaneously passes survivorship, intrinsic dating, SF-dynasty format, and reaches 2021. Label every row's source explicitly as "DynastyProcess FantasyPros-ECR-derived value_2qb."
2. **Decide consciously about the source mismatch.** If the backtest's purpose is "does our model beat *the FantasyCalc market*," DynastyProcess answers a slightly different question ("beat *expert consensus*"). Either (a) accept ECR-consensus as a legitimate market proxy and document it, or (b) treat DynastyProcess as one of two historical baselines and seek a FantasyCalc/KTC market series for the second.
3. **In parallel, probe the two market-based options.** Query the Wayback CDX API for `api.fantasycalc.com/values/current?...numQbs=2...` captures near each date; and inspect the Dynasty Daddy Postgres schema for retained daily KTC-SF rows back to 2021. Either, if it yields clean dated rows, gives you a true market cross-check keyed by Sleeper ID — the exact match to your forward pipeline.
4. **Enforce the integrity gate on ingestion:** reject any row whose `scrape_date`/capture date is more than 7 days from the target or later than the snapshot date; require since-departed players to be present before accepting a snapshot as genuine.
5. **Thresholds that change the plan:** If DynastyProcess lacks a commit within ±7 days of any date, **skip that date** rather than widening the window. If a FantasyCalc Wayback capture with `numQbs=2` exists within ±7 days for a given date, **prefer it** for that date (exact source match to the forward pipeline) and keep DynastyProcess as the corroborating series.

## Caveats
- **Source-methodology mismatch is the biggest risk**, not data integrity: DynastyProcess (ECR-derived) ≠ FantasyCalc (trade-market). This is a feature-definition issue for the backtest and must be documented in results, or a mediocre/strong model could be mis-judged simply because the historical benchmark isn't the same instrument as the forward one.
- **Unverified specifics:** the four exact commit SHAs/dates, the verbatim row samples, the in-file `scrape_date` values, and the literal survivorship name-check were not directly fetched during research (GitHub API/raw access was unavailable to the tools); they are expected to hold based on the repo's documented weekly-Action design and the verified `values.csv` schema, and must be confirmed locally via the git steps.
- **Schema drift over time:** DynastyProcess stabilized column names "as of May 2020" (per its own docs: "I recently renamed some of the columns in the values table (as of May 2020) and don't anticipate these changing again in the near future"), so 2021–2024 schemas should match, but verify the header row at the 2021 commit in case an older file name applies.
- **A missing date is acceptable** per your spec; do not substitute a current snapshot or a non-SF/redraft dataset to fill a gap. FantasyCalc/KTC live endpoints and undated community charts fail the point-in-time test and must not be used for history.
