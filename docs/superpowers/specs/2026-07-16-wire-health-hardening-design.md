## 2. Design

**SHAPE (revised round-6, exercising the pre-named collapse threshold — both lanes concur):** one shared **delivery state-machine module `scripts/dg_delivery.py`** owns send-id creation, the canonical wire-body digest, every lease/claim state transition, attempt/tombstone persistence, locking, and terminal/retryability classification. `scripts/tmux_msg.py` (sender CLI) and `scripts/dg_mail_carrier.py` (carrier CLI) are thin entrypoints DRIVING that one machine — neither holds delivery state or emits a key outside a machine-granted transition. Injectable seams: the existing `Runner` (`scripts/tmux_msg.py:18`), a shared `Capturer`, an injectable clock, and an injectable store, so REDs are hermetic.

**D0 — the delivery state machine [round-6 B1/B2].**
Store: one SQLite file (harness-local, `~/dg-cockpit/delivery.db`; injectable for tests) — one row per send-id carrying `state`, `wire_digest`, `pane_id`, `attempts`, timestamps. **Transitions are serialized CAS updates** (`BEGIN IMMEDIATE` + conditional `UPDATE … WHERE send_id=? AND state=?`; rowcount 1 = transition won, else lost) — atomic-rename file leases are RETIRED (they protect file integrity, not the compound check-then-key transition). States [naming per D0.1, round-9 sweep]: `composing → pasted → composed_verified → submit_attempt(n) → delivered_verified | delivered_content_unconfirmed | delivery_unconfirmed`, refusal states (`refused_*`, `input_not_verifiable`, `input_not_empty`, `manual_clear_required`) which are **terminal and carrier-INELIGIBLE**, and the carrier path `composed_verified → orphan_candidate → orphan_eligible → press_attempt(n) → delivered_* | strand_lost`, with `episode_exhausted` as the tombstone. **Row schema pinned [round-9 B3]:** `send_id, state, wire_digest, pane_id, pane_epoch, profile, baseline_ref (durable pre-send baseline digest), composer_observable (recorded chip marker/count or visible-body digest), attempt counters, per-attempt result evidence (attempt n → timestamp + post-key snapshot digest + resolution), created/updated timestamps` — everything F53's cross-process crash reconciliation consumes is durable in the row, not in process memory. **Pane-ownership rule [round-7 B1]:** send rows alone cannot serialize one physical composer — a `panes` table carries `(pane_id, owner_send_id, epoch)`, and EVERY paste or key additionally requires winning/holding the CURRENT pane claim (epoch CAS); carrier-owning-orphan-A vs sender-creating-B on the same pane resolves to exactly one owner — the loser mutates nothing. **Key-emission rule:** ANY key emission (sender `C-m`, carrier `C-m`) requires currently holding BOTH a won row transition (T5/T5a submit attempts, T12/T12a press attempts) AND the current pane claim; a sender cannot enter `composing` for a strand while a carrier claim owns it, and a carrier cannot claim while the row sits in any sender-live state. **Attempt/crash protocol [round-7 B2]:** every key press is its own claimed attempt (`submit_attempt(n)` for sender incl. the F6 retry, `press_attempt(n)` for carrier within the F22 timeline), each with a recorded claim row and a result transition — SQLite cannot atomically prove the external key fired, so the two crash windows (claim-won-but-no-key vs key-sent-but-no-result-recorded) are reconciled on restart CONSERVATIVELY: unchanged verified input observable → the attempt may be reclaimed within the remaining budget; a NEW verified conversation occurrence of the send-id → resolve delivered; anything ambiguous → terminal/manual — a blind key replay is never a legal reconciliation. **Store failure pins [round-7 H1, injection points per round-8 H2]:** a missing, corrupt, schema-mismatched, or lock-timeout store yields ZERO pane mutations and a named error (`store_unavailable` / `store_schema_mismatch` / `store_locked`) — never silent recreation or state reset. Faults are contract-defined at every point BETWEEN claim and side effect and between side effect and result recording: a pre-side-effect store failure aborts with zero pane mutation; a post-mutation store failure resolves through the conservative restart reconciliation (terminal/manual when ambiguous). SQLite durability pinned: WAL journal, `synchronous=FULL`, `busy_timeout` as an injectable seam, schema version enforced on open. **Store initialization [round-8 B2]:** normal drivers NEVER create or migrate the store; a separate idempotent `dg_delivery.py init-store` command (deployment-gated, in the David-worded sequence) creates the pinned schema at the pinned path with 0600 permissions, refuses to touch an existing non-empty or non-DB file (no overwrite, `init_target_exists`), verifies by read-back, and migrates only with an automatic pre-migration backup + rollback path — **the store is WAL-mode, so the backup is transaction-consistent by contract [round-9 H2]: `VACUUM INTO` (or the SQLite backup API) after a `wal_checkpoint(TRUNCATE)`, never a bare file copy; the backup is read-back verified BEFORE migration begins, and rollback = restore of that verified backup, itself read-back verified.** **Pane-claim lifecycle [round-8 B1]:** claims are state-specific — `delivered_*` with a verified-clear input and every pre-paste-safe refusal RELEASE the pane claim immediately; the blocking manual states (`manual_clear_required`, `input_not_verifiable`, `input_not_empty`, `orphan_manual_required`) RETAIN a blocking claim (no new sender may paste over uncleared text) until a human-side `ack-clear` command proves the same pane identity plus two agreeing READY/EMPTY snapshots, then releases by epoch CAS with zero keys; a stale claim (owner silent past a pinned 300s) recovers per the T16 fork [round-10 H3 sweep]: T16a (row durably `composed_verified`) releases into the carrier handoff with profile atomicity; T16b (every other state) recovers ONLY into a blocking manual state via epoch bump — never into an automated send; an epoch mismatch always loses with zero mutations. **Empty-composer precondition [round-8 B3]:** a pane claim proves ownership, not emptiness — before a FIRST paste the sender additionally requires two agreeing READY snapshots whose ghost-stripped real input is EMPTY; nonempty real input → `input_not_empty` (blocking, zero mutations); dim-only ghost text strips to empty and passes; an unparseable/ambiguous input region blocks as `manual_clear_required`; the ONLY legal nonempty entry is an exact machine resume (an existing row in `pasted`/`composed_verified` whose recorded observable matches the pane).

**D1 — pane-state classifier, debounced.**
`classify_pane(capture_with_escapes: str, profile: PaneProfile) -> PaneState` where `PaneState ∈ {READY, BUSY, DIALOG, UNKNOWN}` and `PaneProfile` is a small per-CLI signature table (claude / codex / gemini: accepting-prompt patterns, busy patterns e.g. `Working (`, dialog patterns e.g. numbered-option permission blocks). `UNKNOWN` is fail-closed: treated as not-ready, never silently sent into. **Debounce law (E3):** a state transition INTO `READY` requires ≥2 consecutive agreeing captures separated by a short gap — the busy indicator flickers mid-run, so a single READY read is never sufficient (a single BUSY/DIALOG read IS sufficient to refuse: asymmetric, conservative direction).

**D2 — readiness gate before send (identity-bound, policy-pinned).**
Target resolution happens ONCE: the user-facing target (`dynasty:1.2`) resolves to the stable tmux `pane_id` (`%N`) plus recorded metadata (title, current_command); every subsequent capture/paste/key op uses the `pane_id`, and a vanished or metadata-mismatched pane (respawn, profile mismatch vs the declared CLI) fails closed (`pane_identity_changed`, `profile_mismatch`) [Codex H3]. `send_message(..., require_ready=True)` refuses unless `READY` (named refusals: `pane_busy`, `pane_dialog`, `pane_state_unknown`, `pane_unreadable`; distinct nonzero exits, enumerated in the CLI help as the result contract). **Pinned timing policy [Codex H2]:** captures 500ms apart; READY = 2 consecutive agreeing reads; `--wait` budget default 30s (attempt count derived), injectable clock/capturer seams so REDs run on fake-clock sequences; capture failure at any point → immediate `pane_unreadable`. **No force path [Codex B5, superseding v4's force-on-UNKNOWN]:** `--force` is REMOVED entirely. UNKNOWN can itself be an unrecognized dialog, so no automated flag may send into it; every non-READY classification refuses, and the escape hatch for profile rot is a human typing in the pane directly (documented in the cockpit-messaging skill) — never agent automation. **Typed results [Codex H3, round-5 H1, round-6 H1]:** the API returns a `SendResult` dataclass (`status`, `reason`, `phase`, `terminal: bool`, `attempts`, `snapshots`, `meta: dict`) where `phase ∈ {pre_paste, pasted, submitted}` records the last pane MUTATION reached — and **terminality follows phase, not status**: every result whose phase is `pasted` or `submitted` is `terminal=True` (a post-Enter `delivery_unconfirmed` may already have delivered; a post-paste `input_not_verifiable` retry would append a second body; neither is safely auto-retryable — manual/human resolution only, unless delivery is positively disproved AND the input is safely restored, which per D6a is itself a human act). Pre-paste refusals are the only auto-retryable class; the named caller boundary the F44 RED exercises is the cockpit-messaging skill's reference send wrapper (`send_and_confirm`), which consults `result.terminal` and has NO resend path when it is true. The CLI maps SendResult to a pinned exit table: `0` delivered-verified · `2` delivery_unconfirmed · `3` refused_not_ready (busy/dialog/unknown, sub-reason in output) · `4` body error · `5` identity/profile error · `6` usage error · `7` delivered_content_unconfirmed (collapsed-chip profiles; semantic receipt required for judgment traffic). **Exit 7 is TERMINAL and non-retryable [round-4 H1]: the bytes already reached the pty, so any automatic resend risks a duplicate delivery — callers wait for semantic receipt or an explicit human resend decision, never retry on 7.**

**D3 — positive post-send verification (two-phase submit, baseline-anchored).**
Verification is defined over a structured `PaneSnapshot` — capture with escapes + cursor/geometry (`capture-pane -e -p` + `display-message` cursor row/col), parsed into **input region** (prompt-anchored lines at/below the cursor line per CLI profile) and **conversation region** (above) — never a whole-pane substring grep [Codex B1]. **Observable contract per CLI profile [Codex round-2 B1 — supersedes v4's full-body pane compare]:** tmux exposes no scrollable-input body, and chip-collapsing CLIs render large pastes as `[Pasted text #N +M lines]` — so what the pane can prove differs by profile and the spec says so: for uncollapsed input, the pane observable is the full visible body (byte compare after ghost-strip + wrap-normalization); for a collapsed chip, the pane observable is the chip marker with its advertised line count, and **body integrity is proven at the buffer layer** — the named per-send buffer (D3a) contains exactly the bytes `paste-buffer` delivers, so buffer content is verified pre-paste and the pane proves presence, not content. **Honesty boundary [round-3 B2]: buffer bytes prove what reached the pty, NOT what the receiving TUI retained** — middle truncation inside the composer is unobservable through a chip. Therefore, post-submit on a collapsed-chip profile: WHEN the conversation region re-exposes the body, it is digest-verified against the canonical wire_body — a mismatch (middle truncation/mangling) is `delivery_unconfirmed`, never delivered; WHEN the body is not re-exposed, the send terminates as `delivered_content_unconfirmed` (exit 7, distinct from verified `0`), and the cockpit-messaging discipline requires SEMANTIC RECEIPT (reply or ledger read-back) before any judgment-bearing content-unconfirmed send is treated as received — the existing ledger-first law, now wired to a named tool status. **Correlation marker [round-3 B3; grammar revised round-6 B3]:** every send stamps a short visible send-id suffixed to the sender header line — `[w#<nonce8>-<seq>]` where `nonce8` is a per-process random base36 nonce from `os.urandom` (PID retired: PID reuse + a reset counter can collide with a tombstone inside the ≥24h/512 retention horizon), collision-bounded across processes AND across the tombstone lifetime. **Body vocabulary pinned:** `original_body` = caller-supplied bytes; `wire_body` = original_body with the send-id stamped onto its first line by a defined transformation; every byte-exact claim (F18/F30) is about original_body content surviving inside wire_body — the stamp is the one sanctioned delta. The first line MUST match the 02 sender-header form (`From <sender>`); a body without it is refused `body_missing_sender_header` (the stamp needs a header line to ride, and 02 requires one anyway). **Snapshot stability:** capture and cursor metadata are fetched in one chained tmux dispatch and a verification read requires two agreeing composite snapshots (atomicity is not guaranteed and is not assumed; mixed-frame reads retry, then fail closed). **D3a — buffer hygiene [Codex B2].**
Every send uses a unique named tmux buffer (`dg-msg-<pid>-<counter>`), pastes explicitly by name (`paste-buffer -b`), and deletes the buffer on ALL exit paths (success, refusal, exception). The user's default/most-recent buffer is never written: concurrent senders each paste their own named buffer (interleaved-sender RED), and a user's copy-paste stack is never clobbered.

**D4 — dim-aware capture helper (stateful SGR contract) [Codex H1].**
`strip_ghost(raw: str) -> str` runs a per-line SGR state machine with **sequential left-to-right parameter semantics [Codex H2]**: parameters in one sequence apply in order, so `\x1b[0;2m` ends DIM-SET (reset, then dim) while `\x1b[2;0m` ends RESET; an EMPTY parameter list (`\x1b[m`) is SGR 0 (reset); dim is SET by parameter 2 at any position, CLEARED by a later 0 or 22 (same or later sequence); state does not leak across lines (tmux capture re-emits attributes per line); a malformed or unclosed dim span dims the REST OF THAT LINE (conservative: dim-until-proven-real). Ghost content is replaced by a single non-matchable placeholder (`�`), never deleted, so stripping cannot concatenate two real fragments into a false whole-body match. All D3 matching runs over `strip_ghost` output. Exposed as `tmux_msg.py capture <target> --no-ghost`; the **ledger-first verdict rule remains the binding gate regardless** (this reduces spoof surface; it does not replace the rule).

**D8 — message bodies via stdin/file ONLY (Codex round-2 B2, superseding v4's threshold rule).**
